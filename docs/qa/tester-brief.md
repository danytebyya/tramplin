# ТЗ для тестировщика

## Цель
Проверить, что текущая версия проекта `Трамплин`:
- проходит базовую техническую проверку;
- визуально соответствует актуальному варианту в Figma;
- не содержит явных регрессий в ключевых пользовательских сценариях.

## Что сделать в начале
1. Поднять проект локально по инструкции из `README.md`.
2. Перед ручной проверкой обязательно запустить все backend-тесты.

Команды:

```bash
cd backend
source .venv/bin/activate
pytest
```

## Что проверять
1. Backend tests:
   Убедиться, что все тесты проходят без падений.

2. Визуальное соответствие Figma:
   Сравнить реализованные страницы с актуальным макетом в Figma.

3. Ключевые UI-проверки:
   Проверить сетку, отступы, размеры элементов, тексты, состояния hover/focus/error/disabled, адаптивность, скролл, поведение форм и интерактивных контролов.

4. Основные пользовательские сценарии:
   Проверить регистрацию, вход, подтверждение email, а также уже доступные сценарии онбординга работодателя и другие реализованные экраны.

## Как фиксировать замечания
Если найдена несостыковка, нужно записать:

- где найдено:
  страница, блок, URL;
- тип проблемы:
  визуальная, функциональная, текстовая, валидация, адаптив, браузерная;
- ожидаемое поведение:
  как в Figma или как должно работать по сценарию;
- фактическое поведение:
  что происходит сейчас;
- шаги воспроизведения;
- браузер и устройство;
- скриншот, если проблема визуальная.

## Формат отчета
Для каждого бага:

```md
### Заголовок
- Страница: ...
- URL: ...
- Браузер: ...
- Тип: ...
- Шаги:
  1. ...
  2. ...
  3. ...
- Ожидаемо: ...
- Фактически: ...
- Ссылка на Figma / комментарий по макету: ...
- Вложение: скриншот
```

## Важно
- Если backend-тесты не запускаются, это тоже нужно зафиксировать как блокирующую проблему.
- Если экран расходится с Figma, нужно писать конкретно: что именно не совпадает.
- Если поведение неочевидно, не додумывать, а отмечать как вопрос на уточнение.
