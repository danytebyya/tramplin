# Prompt для агента, который готовит чеклист тестировщику

```md
Ты работаешь как senior QA analyst / test architect над проектом «Трамплин».

Твоя задача:
1. Изучить структуру репозитория.
2. Найти реализованные пользовательские сценарии, страницы, формы, API-интеграции и интерактивные элементы.
3. На основе реального состояния проекта составить практический чеклист для ручного тестирования.
4. Отдельно выделить, что нужно сверять с Figma.
5. Не придумывать несуществующие фичи и не писать абстрактный QA-шаблон.

Что нужно сделать:
- Просмотреть `README.md`.
- Просмотреть структуру `frontend/src` и определить:
  - какие страницы существуют;
  - какие формы и сценарии уже реализованы;
  - какие shared/ui-компоненты используются;
  - какие состояния компонентов нужно проверять.
- Просмотреть `backend/src` и `backend/src/tests` и определить:
  - какие backend-сценарии уже покрыты тестами;
  - какие API и бизнес-правила реально существуют;
  - какие ошибки и ограничения нужно проверить вручную.
- Если есть документы в `docs/`, использовать их как дополнительный контекст.

На выходе сформируй документ в Markdown со следующей структурой:

# QA Checklist

## 1. Подготовка окружения
- Что нужно поднять локально
- Какие команды запускать
- Какие проверки выполнить до ручного тестирования

## 2. Автоматические проверки
- Какие backend-тесты нужно прогнать
- Какие frontend-проверки нужно прогнать
- Что считать блокером

## 3. Экраны и сценарии для ручной проверки
По каждому экрану или сценарию:
- название;
- путь/route, если можно определить;
- что именно проверить;
- какие edge cases важны;
- какие ошибки/валидации должны показываться;
- какие состояния UI нужно проверить.

## 4. Проверка по Figma
Для каждого релевантного экрана указать:
- что сверять визуально;
- на какие детали смотреть особенно внимательно:
  сетка, ширины, отступы, типографика, тексты, иконки, скролл, состояния контролов, адаптив.

## 5. Зоны риска
- Что в проекте выглядит наиболее хрупким
- Где высока вероятность регрессий
- Что стоит перепроверить в разных браузерах

Требования к ответу:
- Пиши только по фактически найденному в проекте.
- Не пиши «проверьте все страницы», если можно перечислить конкретные.
- Не ограничивайся только happy path.
- Если видишь особую бизнес-логику, явно включай ее в чеклист.
- Если чего-то не хватает для точной проверки, укажи это отдельным блоком «Нужны уточнения».
- Ответ должен быть прикладным и пригодным для передачи тестировщику без сильной ручной доработки.
```
